# API Specification — NestJS Reusable Template

## Overview

| Setting | Value |
|---------|-------|
| Base URL | `/api/v1` |
| Versioning | URL path; breaking changes introduce `/api/v2` |
| Content-Type | `application/json` for normal requests |
| File upload | `multipart/form-data` |
| Protocol | HTTPS in production |
| Encoding | UTF-8 |
| API docs | `GET /docs` for Swagger UI, usually disabled or restricted in production |

This document defines a neutral API contract pattern for NestJS services. Replace example resource names with real domain names only where necessary.

## Authentication

### Token Flow

```text
POST /api/v1/auth/login
  → Response body: { accessToken, user }
  → Optional Set-Cookie: refreshToken=<value>; HttpOnly; Secure; SameSite=Strict
```

| Token | Recommended lifetime | Storage | Delivery |
|-------|----------------------|---------|----------|
| Access token | 5–30 minutes | Client memory or secure platform storage | `Authorization` header |
| Refresh token | 7–30 days | httpOnly cookie or secure platform storage | Cookie or refresh request body, depending on client type |

### Request Header Format

```http
Authorization: Bearer <access_token>
```

### Refresh Token Flow

```text
POST /api/v1/auth/refresh
  → Server validates refresh token hash, expiry, and revoked state
  → Server rotates refresh token if rotation is enabled
  → Response: { accessToken } and optionally new refreshToken cookie
```

Recommended behavior:

- Store only a hash of the refresh token server-side.
- Rotate refresh tokens on every refresh request.
- Revoke all active refresh tokens for the same user if refresh token reuse is detected.
- Use short-lived access tokens.

### Auth Error Codes

| Code | HTTP | Description |
|------|------|-------------|
| `AUTH_001` | 401 | Invalid credentials |
| `AUTH_002` | 401 | Access token expired |
| `AUTH_003` | 401 | Access token invalid or malformed |
| `AUTH_004` | 401 | Refresh token expired, revoked, or invalid |
| `AUTH_005` | 403 | Insufficient permissions |
| `AUTH_006` | 409 | Account already exists |
| `AUTH_007` | 429 | Too many authentication attempts |

### Route Access Matrix

| Access Level | Mechanism | Intended caller |
|--------------|-----------|-----------------|
| Public | No authentication required | Anyone |
| Protected | Valid access token required | Authenticated user |
| Role-based | Protected + required role | Admin, manager, staff, or similar role |
| Ownership-based | Protected + owner/member check | User who owns or belongs to the resource |
| Service-to-service | API key, signed request, mTLS, or internal token | Trusted internal systems |

## Request Conventions

### Pagination

Offset-based pagination:

```http
GET /api/v1/resources?page=2&limit=20
```

| Param | Type | Default | Max | Notes |
|-------|------|---------|-----|-------|
| `page` | integer | 1 | — | 1-indexed |
| `limit` | integer | 20 | 100 | Items per page |

Cursor-based pagination, when required for large or fast-changing lists:

```http
GET /api/v1/resources?cursor=eyJpZCI6MTAwfQ&limit=20
```

Use one pagination style per endpoint. Do not mix offset and cursor in the same endpoint.

### Sorting

```http
GET /api/v1/resources?sort=created_at&order=desc
```

| Param | Values | Default |
|-------|--------|---------|
| `sort` | Whitelisted field per endpoint | `created_at` |
| `order` | `asc` or `desc` | `desc` |

Rules:

- Every sortable field must be whitelisted server-side.
- Do not pass raw sort values directly into SQL.
- Use stable secondary ordering when pagination requires deterministic results.

### Filtering

```http
GET /api/v1/resources?status=active&type=internal
GET /api/v1/resources?created_from=2026-01-01&created_to=2026-01-31
GET /api/v1/resources?search=keyword
```

Rules:

- Define allowed filters per endpoint.
- Validate filter types using query DTOs.
- Use ISO 8601 dates for date/time filters.
- Prefer explicit filter names over overloaded generic parameters.

### File Upload

```http
Content-Type: multipart/form-data
```

| Setting | Recommended default |
|---------|---------------------|
| Single file field | `file` |
| Multiple files field | `files` |
| Max file size | Project-defined, commonly 5–20 MB |
| Allowed MIME types | Whitelist per endpoint |
| Storage | Local, S3-compatible, or project storage adapter |

Rules:

- Validate MIME type and file extension.
- Generate server-side file names.
- Do not trust client-provided file names.
- Store metadata in the database if files are part of the domain model.

## Response Format

All responses should be wrapped by a global transform interceptor.

### Success — Single Resource

```json
{
  "success": true,
  "data": {
    "id": 1,
    "name": "Sample resource"
  },
  "message": "Created successfully"
}
```

### Success — List

```json
{
  "success": true,
  "data": [
    { "id": 1 },
    { "id": 2 }
  ]
}
```

### Success — Paginated List

```json
{
  "success": true,
  "data": [
    { "id": 1 },
    { "id": 2 }
  ],
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 95,
    "totalPages": 5
  }
}
```

### Success — No Content

For `204 No Content`, return an empty response body.

### Error

```json
{
  "success": false,
  "error": {
    "code": "RESOURCE_001",
    "message": "Resource not found",
    "details": {}
  }
}
```

### Validation Error

```json
{
  "success": false,
  "error": {
    "code": "SYS_002",
    "message": "Validation failed",
    "details": {
      "email": "must be a valid email address",
      "name": "should not be empty"
    }
  }
}
```

## Error Code Reference

### Format

```text
{FEATURE_PREFIX}_{3-digit sequential number}
```

Examples:

| Prefix | Meaning |
|--------|---------|
| `AUTH` | Authentication and authorization |
| `USER` | User/account related errors |
| `RESOURCE` | Generic resource feature errors; rename per project |
| `WORKFLOW` | Multi-step operation errors; rename per project |
| `FILE` | Upload/storage errors |
| `SYS` | System-level errors |

### Common Error Codes

| Code | HTTP | Description |
|------|------|-------------|
| `AUTH_001` | 401 | Invalid credentials |
| `AUTH_002` | 401 | Access token expired |
| `AUTH_003` | 401 | Access token invalid |
| `AUTH_004` | 401 | Refresh token expired or revoked |
| `AUTH_005` | 403 | Insufficient permissions |
| `AUTH_006` | 409 | Account already exists |
| `USER_001` | 404 | User not found |
| `USER_002` | 400 | Invalid current password |
| `RESOURCE_001` | 404 | Resource not found |
| `RESOURCE_002` | 409 | Resource already exists |
| `RESOURCE_003` | 400 | Invalid resource state |
| `RESOURCE_004` | 403 | Resource does not belong to current user or workspace |
| `WORKFLOW_001` | 400 | Workflow cannot be executed in current state |
| `WORKFLOW_002` | 409 | Workflow was already processed |
| `FILE_001` | 400 | File type not allowed |
| `FILE_002` | 400 | File size exceeds limit |
| `FILE_003` | 500 | File storage failed |
| `SYS_001` | 500 | Internal server error |
| `SYS_002` | 400 | Validation failed |
| `SYS_003` | 429 | Rate limit exceeded |

### HTTP Status Mapping

| Status | Use when |
|--------|----------|
| 200 | Successful read or update |
| 201 | Successful creation |
| 202 | Request accepted for asynchronous processing |
| 204 | Successful deletion or action with no response body |
| 400 | Invalid input or business rule violation |
| 401 | Authentication required or failed |
| 403 | Authenticated but not authorized |
| 404 | Resource not found |
| 409 | Duplicate or state conflict |
| 422 | Semantically invalid request where validation passed but domain meaning is invalid |
| 429 | Rate limit exceeded |
| 500 | Unhandled server error |
| 503 | Temporary service dependency failure |

## Endpoint Details

### POST `/auth/register`

Request:

```json
{
  "email": "user@example.com",
  "password": "securePassword123",
  "fullName": "Example User"
}
```

Validation:

| Field | Rule |
|-------|------|
| `email` | Required, valid email, max 255 chars |
| `password` | Required, minimum 8 chars or project password policy |
| `fullName` | Required or optional depending on project rules, max 100 chars |

Response `201`:

```json
{
  "success": true,
  "data": {
    "id": 1,
    "email": "user@example.com",
    "fullName": "Example User",
    "role": "user"
  },
  "message": "Registration successful"
}
```

Errors:

| Error | Condition |
|-------|-----------|
| `AUTH_006` | Account already exists |
| `SYS_002` | Invalid request fields |

### PATCH `/resources/:id/status`

Request:

```json
{
  "status": "inactive",
  "reason": "No longer needed"
}
```

Response `200`:

```json
{
  "success": true,
  "data": {
    "id": 1,
    "status": "inactive",
    "updatedAt": "2026-01-15T11:00:00.000Z"
  },
  "message": "Status updated successfully"
}
```

Business logic:

1. Verify resource exists.
2. Verify actor has permission.
3. Verify status transition is allowed.
4. Save status update.
5. Emit `resource.status_changed` event if downstream side effects are needed.

Errors:

| Error | Condition |
|-------|-----------|
| `RESOURCE_001` | Resource not found |
| `RESOURCE_003` | Invalid state transition |
| `RESOURCE_004` | Actor has no access to this resource |

## NestJS Implementation Notes

### Protected Controller Pattern

```ts
@ApiTags('resources')
@ApiBearerAuth()
@Controller('resources')
@UseGuards(JwtAuthGuard)
export class ResourceController {
  constructor(private readonly resourceService: ResourceService) {}

  @Post()
  @HttpCode(HttpStatus.CREATED)
  @ApiOperation({ summary: 'Create resource' })
  @ApiResponse({ status: 201, type: ResourceResponseDto })
  async create(
    @CurrentUser() user: IAuthPayload,
    @Body() dto: CreateResourceDto,
  ): Promise<ResourceResponseDto> {
    return this.resourceService.create(user.id, dto);
  }
}
```

### Public Endpoint Pattern

```ts
@ApiTags('resources')
@Controller('resources')
export class PublicResourceController {
  constructor(private readonly resourceService: ResourceService) {}

  @Public()
  @Get()
  async findAll(@Query() query: QueryResourceDto): Promise<PaginatedResponse<ResourceResponseDto>> {
    return this.resourceService.findAll(query);
  }
}
```

### Rate Limiting Pattern

```ts
@Throttle({ default: { limit: 5, ttl: 60_000 } })
@Post('login')
async login(@Body() dto: LoginDto): Promise<LoginResponseDto> {
  return this.authService.login(dto);
}
```

### Transactional Action Pattern

```ts
@Post(':id/action')
@UseGuards(JwtAuthGuard)
async executeAction(
  @CurrentUser() user: IAuthPayload,
  @Param('id', ParseIntPipe) id: number,
  @Body() dto: ExecuteActionDto,
): Promise<ActionResponseDto> {
  return this.resourceActionService.execute(user.id, id, dto);
}
```

## Reuse Checklist

When adapting this API spec to a concrete project:

1. Replace `resources` with the actual domain resource names.
2. Replace `RESOURCE_*` and `WORKFLOW_*` prefixes with feature-specific prefixes.
3. Decide which endpoints are public, protected, role-based, ownership-based, or internal.
4. Define endpoint-specific filters and sortable fields.
5. Define file upload limits and allowed MIME types.
6. Define token storage strategy for each client type.
7. Add domain-specific endpoint details only where required.
