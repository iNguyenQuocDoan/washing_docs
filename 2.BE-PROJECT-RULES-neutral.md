# Backend Project Rules — NestJS Reusable Template

## Purpose

This document defines backend engineering rules for a reusable NestJS project structure. It is intentionally domain-neutral, so it can be reused across projects such as admin systems, marketplaces, SaaS products, booking systems, internal tools, or content platforms with minimal or no rewriting.

Use domain-specific names only inside actual feature folders, DTOs, entities, and endpoint definitions.

---

## Tech Stack

| Technology | Recommended Version | Role |
|------------|---------------------|------|
| Language | TypeScript 5.x, strict mode | Application language |
| Framework | NestJS 11.x | HTTP framework and dependency injection container |
| ORM | TypeORM | Database access layer |
| Database | MySQL 8.x or compatible SQL database | Primary relational datastore |
| Validation | class-validator + class-transformer | DTO validation and transformation |
| Testing | Jest | Unit and integration tests |
| API docs | Swagger / OpenAPI | API documentation |
| Auth | Passport + JWT | Authentication and authorization |

> If a project uses PostgreSQL, Prisma, MikroORM, Redis, queues, or object storage, keep the same architecture rules and replace only the infrastructure adapters.

## 1. Project Layout

### Top-Level Layout

```text
src/
├── main.ts                         # Bootstrap: global pipes, filters, interceptors, docs
├── app.module.ts                   # Root module: imports config, core, shared, features
│
├── config/                         # Typed environment config; no raw process.env outside here
│   ├── app.config.ts
│   ├── database.config.ts
│   ├── auth.config.ts
│   └── storage.config.ts           # Optional: files, S3, CDN, local storage
│
├── core/                           # Infrastructure plumbing; no domain business logic
│   ├── database/
│   │   └── database.module.ts      # TypeORM forRootAsync setup
│   ├── logger/
│   │   └── logger.module.ts        # Logger setup
│   ├── cache/                      # Optional: Redis/cache module
│   ├── queue/                      # Optional: background job module
│   └── storage/                    # Optional: file/object storage adapter
│
├── shared/                         # Cross-feature reusable code
│   ├── decorators/                 # @CurrentUser(), @Roles(), @Public()
│   ├── filters/                    # Global exception filters
│   ├── guards/                     # Auth and authorization guards
│   ├── interceptors/               # Response transform, logging, timeout
│   ├── pipes/                      # Validation and parsing pipes
│   ├── utils/                      # Pagination, hashing, date, string helpers
│   ├── constants/                  # Shared constants only
│   └── types/                      # ApiResponse<T>, PaginationMeta, AuthPayload
│
└── features/
    ├── auth/                       # Authentication, token lifecycle, identity guards
    ├── account/                    # Optional: user/account profile data
    ├── resource-a/                 # Replace with a real domain feature
    ├── resource-b/                 # Replace with a real domain feature
    └── audit-log/                  # Optional: audit trail / system activity
```

### Feature Internal Structure

Use the flat structure for simple features:

```text
features/[feature-name]/
├── [feature].module.ts             # Declares providers, imports, exports
├── [feature].controller.ts         # Route handlers and DTO binding
├── [feature].service.ts            # Business logic and orchestration
├── repositories/
│   └── [entity].repository.ts      # TypeORM queries only
├── dto/
│   ├── create-[entity].dto.ts
│   ├── update-[entity].dto.ts
│   ├── query-[entity].dto.ts
│   └── [entity]-response.dto.ts
├── entities/
│   └── [entity].entity.ts          # TypeORM entity definitions
├── types/
│   └── [feature].types.ts          # Feature-local interfaces and types
├── events/                         # Optional: emitted/listened domain events
├── tests/
│   ├── [feature].service.spec.ts
│   ├── [feature].controller.spec.ts
│   └── [entity].repository.spec.ts
└── CONTEXT.md                      # Optional AI/project context for this feature
```

Use grouped subfolders for complex features:

```text
features/[feature-name]/
├── [feature].module.ts
├── controllers/
├── services/
├── repositories/
├── entities/
├── dto/
├── types/
├── events/
├── tests/
└── CONTEXT.md
```

## 2. Naming Conventions

| Element | Convention | Example |
|---------|------------|---------|
| Feature folders | kebab-case | `account/`, `audit-log/`, `resource-management/` |
| Files | kebab-case + layer suffix | `create-resource.dto.ts`, `resource.entity.ts`, `resource.service.ts` |
| Classes | PascalCase | `ResourceService`, `CreateResourceDto` |
| Functions / methods | camelCase | `findById()`, `createResource()`, `hashToken()` |
| Variables | camelCase | `resourceId`, `currentUser`, `tokenHash` |
| Constants | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE`, `DEFAULT_LIMIT` |
| Interfaces | PascalCase with `I` prefix | `IAuthPayload`, `IRequestContext` |
| Type aliases | PascalCase with `Type` suffix | `ResourceStatusType` |
| Entities | PascalCase, singular | `User`, `Resource`, `AuditLog` |
| DTOs | PascalCase with `Dto` suffix | `CreateResourceDto`, `LoginResponseDto` |
| Enums | PascalCase with `Enum` suffix | `ResourceStatusEnum`, `RoleEnum` |

**DO vs DON'T**

| Do not | Do |
|--------|----|
| `getuser.ts` | `user.service.ts` |
| `createresource()` | `createResource()` |
| `RESOURCESERVICE` | `ResourceService` |
| `authPayload` for an interface | `IAuthPayload` |
| `status` for reusable status type alias | `ResourceStatusType` |

## 3. Feature Isolation Rules

### Ownership

Each feature owns its entities, repositories, DTOs, and business rules. No feature may directly import another feature's internal repository or entity unless that dependency is explicitly exported through the owning module.

| Feature Type | Owns |
|--------------|------|
| `auth` | Identity, credential verification, token lifecycle, role/permission checks |
| `account` | Profile data and account-level preferences |
| Domain feature | Its entities, repositories, DTOs, commands, queries, and business rules |
| `audit-log` | System activity records and append-only audit trail |
| `notification` | Notification templates, delivery adapters, and send logs |

### Allowed Cross-Feature Communication

```ts
// Pattern 1: Module import + DI for synchronous calls
@Module({
  imports: [ResourceModule],
  providers: [WorkflowService],
})
export class WorkflowModule {}
```

```ts
// Pattern 2: EventEmitter2 for async side effects
this.eventEmitter.emit('resource.created', {
  resourceId: resource.id,
  actorId: currentUser.id,
});
```

```ts
// Pattern 3: shared service only when the logic is genuinely reusable by 3+ features
@Injectable()
export class SlugService {
  createSlug(input: string): string {
    return normalizeSlug(input);
  }
}
```

Forbidden:

```ts
// Do not import another feature's internal implementation directly
import { ResourceRepository } from '../resource/repositories/resource.repository';
import { ResourceEntity } from '../resource/entities/resource.entity';
```

### Choosing a Communication Pattern

| Pattern | Use when |
|---------|----------|
| Module import + DI | A feature needs a synchronous method exposed by another feature |
| EventEmitter2 | The next action is a decoupled side effect such as notifications, audit logs, indexing, analytics |
| Shared utility/service | The logic is generic and used by at least 3 unrelated features |

## 4. Mandatory Code Patterns

### Controllers

Controllers only handle routing, parameter extraction, DTO binding, guard/decorator usage, and response status codes. They must not contain business logic.

```ts
@ApiTags('resources')
@Controller('resources')
export class ResourceController {
  constructor(private readonly resourceService: ResourceService) {}

  @Post()
  @UseGuards(JwtAuthGuard)
  async create(
    @CurrentUser() user: IAuthPayload,
    @Body() dto: CreateResourceDto,
  ): Promise<ResourceResponseDto> {
    return this.resourceService.create(user.id, dto);
  }
}
```

### Services

Services own business rules, orchestration, transactions, and cross-feature coordination.

```ts
@Injectable()
export class ResourceService {
  private readonly logger = new Logger(ResourceService.name);

  constructor(private readonly resourceRepository: ResourceRepository) {}

  async create(actorId: number, dto: CreateResourceDto): Promise<Resource> {
    this.logger.log('Creating resource', { actorId });

    const exists = await this.resourceRepository.existsByName(dto.name);
    if (exists) {
      throw new ConflictException('Resource already exists');
    }

    return this.resourceRepository.createAndSave({ ...dto, createdById: actorId });
  }
}
```

### DTO Validation

All external input must be validated at the controller boundary using DTOs.

```ts
export class CreateResourceDto {
  @ApiProperty({ example: 'Sample resource' })
  @IsString()
  @IsNotEmpty()
  @MaxLength(255)
  name: string;

  @IsOptional()
  @IsString()
  @MaxLength(1000)
  description?: string;

  @IsOptional()
  @IsBoolean()
  @Type(() => Boolean)
  isActive?: boolean;
}
```

Global validation pipe:

```ts
app.useGlobalPipes(
  new ValidationPipe({
    transform: true,
    whitelist: true,
    forbidNonWhitelisted: true,
  }),
);
```

### Repository Pattern

Repositories contain TypeORM access and query builders only. They must not contain business policy.

```ts
@Injectable()
export class ResourceRepository {
  constructor(
    @InjectRepository(Resource)
    private readonly repo: Repository<Resource>,
  ) {}

  async findActiveById(id: number): Promise<Resource | null> {
    return this.repo.findOne({ where: { id, isActive: true } });
  }

  async existsByName(name: string): Promise<boolean> {
    return this.repo.exists({ where: { name } });
  }
}
```

### Error Handling

Use NestJS HTTP exceptions for common cases and custom domain exceptions for reusable structured errors.

```ts
throw new NotFoundException('Resource not found');
throw new BadRequestException('Invalid state transition');
throw new ConflictException('Resource already exists');
throw new ForbiddenException('You do not have access to this resource');
```

```ts
export class DomainRuleViolationException extends HttpException {
  constructor(code: string, message: string, details?: Record<string, unknown>) {
    super(
      { code, message, details: details ?? {} },
      HttpStatus.BAD_REQUEST,
    );
  }
}
```

Never swallow errors:

```ts
try {
  await this.performOperation();
} catch (error) {
  this.logger.error('Operation failed', { error: error.message });
  throw error;
}
```

### Response Format

Controllers return raw data. A global `TransformInterceptor` wraps successful responses.

```json
{
  "success": true,
  "data": { "id": 1, "name": "Sample resource" },
  "message": "Created successfully"
}
```

Paginated response:

```json
{
  "success": true,
  "data": [],
  "meta": { "page": 1, "limit": 20, "total": 0, "totalPages": 0 }
}
```

Error response:

```json
{
  "success": false,
  "error": {
    "code": "RESOURCE_001",
    "message": "Resource not found",
    "details": {}
  }
}
```

### Logging

Use `Logger` or the project logger at the service layer. Include structured context.

```ts
this.logger.log('Resource updated', {
  resourceId,
  actorId,
  operation: 'updateResource',
});
```

Log error context, then rethrow:

```ts
this.logger.error('Resource update failed', {
  resourceId,
  actorId,
  error: error.message,
});
throw error;
```

### Transactions

Use `QueryRunner` for multi-step write operations that must succeed or fail as a single unit.

```ts
const queryRunner = this.dataSource.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction();

try {
  // 1. Validate current state
  // 2. Write primary record
  // 3. Write related records
  // 4. Insert audit log or outbox record if needed
  await queryRunner.commitTransaction();
} catch (error) {
  await queryRunner.rollbackTransaction();
  throw error;
} finally {
  await queryRunner.release();
}
```

## 5. Anti-Patterns

| Do not | Do | Reason |
|--------|----|--------|
| Import another feature's internal repository/entity directly | Use module exports, public services, or events | Preserves module boundaries |
| Put business logic in controllers | Keep business logic in services | Controllers remain thin and testable |
| Put query-builder logic in services | Use repositories | Separates business rules from persistence |
| Use raw `process.env` in feature code | Use typed config and `ConfigService` | Centralized, testable configuration |
| Store plain secrets or passwords | Hash using a strong algorithm | Security requirement |
| Use `any` | Use explicit types, generics, or `unknown` with narrowing | Preserves TypeScript safety |
| Create circular module dependencies | Refactor boundary or use events | Avoids NestJS bootstrap failures |
| Silent catch blocks | Log context and rethrow, or convert to a known exception | Prevents hidden production failures |
| Return inconsistent response shapes | Use global response/error interceptors and filters | Improves client integration |
| Hardcode domain-specific constants in shared code | Keep domain constants inside feature modules | Keeps shared code reusable |

## 6. Git Workflow

### Branch Naming

```text
[type]/[scope]-[short-description]

feature/auth-refresh-token-rotation
fix/resource-status-transition
refactor/audit-log-listener
chore/update-dependencies
test/resource-service
```

### Commit Messages

Use Conventional Commits:

```text
[type]([scope]): [description]

feat(auth): add refresh token rotation
fix(resource): validate status before update
refactor(core): extract database config
test(account): add service unit tests
docs(api): update error response format
```

Allowed types: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `perf`, `ci`, `build`.

### Pull Request Requirements

| Requirement | Detail |
|-------------|--------|
| Scope | One feature, fix, or refactor per PR |
| Tests | Relevant unit and integration tests pass |
| Type-check | `tsc --noEmit` passes |
| Lint | ESLint passes without errors |
| Migration review | Database migration is included and reviewed when schema changes |
| API docs | Swagger/API spec updated for public contract changes |
| Context docs | Feature `CONTEXT.md` updated when behavior or public API changes |
| Review | At least one reviewer approval |

## 7. Testing Rules

### Coverage Targets

| Layer | Target | Location |
|-------|--------|----------|
| Services | At least 80% | `features/[feature]/tests/[feature].service.spec.ts` |
| Controllers | At least 70% | `features/[feature]/tests/[feature].controller.spec.ts` |
| Repositories | At least 60% | `features/[feature]/tests/[entity].repository.spec.ts` |

### Test Structure

Use Arrange → Act → Assert.

```ts
describe('ResourceService', () => {
  let service: ResourceService;
  let resourceRepository: jest.Mocked<ResourceRepository>;

  beforeEach(async () => {
    const module = await Test.createTestingModule({
      providers: [
        ResourceService,
        {
          provide: ResourceRepository,
          useValue: {
            existsByName: jest.fn(),
            createAndSave: jest.fn(),
          },
        },
      ],
    }).compile();

    service = module.get(ResourceService);
    resourceRepository = module.get(ResourceRepository);
  });

  describe('create', () => {
    it('creates a resource when name is unique', async () => {
      resourceRepository.existsByName.mockResolvedValue(false);
      resourceRepository.createAndSave.mockResolvedValue(mockResource);

      const result = await service.create(actorId, createResourceDto);

      expect(result).toEqual(mockResource);
      expect(resourceRepository.createAndSave).toHaveBeenCalledWith(
        expect.objectContaining({ createdById: actorId }),
      );
    });

    it('throws ConflictException when name already exists', async () => {
      resourceRepository.existsByName.mockResolvedValue(true);

      await expect(service.create(actorId, createResourceDto)).rejects.toThrow(ConflictException);
    });
  });
});
```

### What to Test

| Layer | Test focus |
|-------|------------|
| Controller | Guards, route binding, DTO usage, expected service call |
| Service | Business rules, orchestration, errors, events, transaction boundaries |
| Repository | Query correctness, filters, pagination, joins, locking when required |
| E2E | Critical flows across HTTP boundary |

## 8. NestJS Decorators Quick Reference

```ts
// Custom decorators
@CurrentUser()          // Extracts authenticated user from request context
@Roles('admin')         // Adds role metadata for RolesGuard
@Public()               // Marks endpoint as public

// Guards
@UseGuards(JwtAuthGuard)
@UseGuards(JwtAuthGuard, RolesGuard)

// Rate limiting
@Throttle({ default: { limit: 5, ttl: 60_000 } })
@Post('login')
async login(@Body() dto: LoginDto) {}

// Swagger
@ApiTags('resources')
@ApiOperation({ summary: 'Create resource' })
@ApiResponse({ status: 201, type: ResourceResponseDto })
@ApiBearerAuth()

// HTTP behavior
@HttpCode(HttpStatus.NO_CONTENT)
@Delete(':id')
async remove(@Param('id', ParseIntPipe) id: number) {}
```

## 9. Reuse Checklist

Before using this document in a new project, only change these items if needed:

- Replace example feature names such as `resource-a`, `resource-b`, `account`, or `audit-log` with real domain features.
- Adjust database choice if the project does not use MySQL or TypeORM.
- Adjust authentication strategy if the project does not use JWT.
- Adjust optional infrastructure modules such as cache, queue, storage, and notification.
- Define domain-specific error code prefixes in the API specification.
